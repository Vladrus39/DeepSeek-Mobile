## In-app full-agent project creation (live, driven from chat)

These runs were performed by typing a request into the app's chat on the device (not by ADB scripts), exercising the real agent tool-calling loop against the Termux executor.

| Project | What the agent did | Observed output |
|---------|--------------------|-----------------|
| `demo_app` | Created folder, wrote `run.sh` (Fibonacci), `chmod +x`, ran it, then `git init` + commit | Real Fibonacci sequence `0,1,1,2,3,5,вЂ¦`; git commit created |
| `calc_app` | Wrote `calc.py` (add/subtract/multiply/divide + main), `test_calc.py` with assert tests, ran the program and the tests, then `git init` + commit | `add(10,5)=15`, `subtract(10,5)=5`, `multiply(10,5)=50`, `divide(10,5)=2.0`; tests pass; commit with 2 files / 55 insertions |

The agent generated correct shell (heredoc file creation, `&&`-chained commands) and produced **differentiated, correct program output** вЂ” evidence of real tool execution, not chat text.

**Verification boundary (Android sandboxing).** The agent works in Termux's private home `/data/data/com.termux/files/home/`. This directory is **not** readable by `adb` (Termux is not a debuggable app), Android 11+ scoped storage blocks Termux from writing to `/sdcard` (confirmed: even after `pm grant вЂ¦_EXTERNAL_STORAGE`, a Termux copy to `/sdcard/dsproof` does not appear to `adb`), and the app's Files tab reads a separate app-private workspace dir. There is therefore no external (adb) porthole into Termux home on this device; the created files are confirmed in-app via the Termux view and the agent's live command output.

## PC Host live file creation (externally verified on disk)

Unlike the phone's Termux sandbox, the PC Host writes into an ordinary Windows directory, so the created files are **directly verifiable from outside the process**. Run `ds-verify.bat` (which calls `scripts/pc-host-e2e.ps1`): it launches `deepseek-pc-host.exe` against a throwaway workspace, drives the real gateway HTTP protocol (`POST /v1/gateway/request`), then reads the files back off disk.

| Gateway request | Result (2026-05-29) |
|-----------------|---------------------|
| `WriteFile hello.py` | `FileWritten { bytes: 36 }`; file present on disk |
| `WriteFile pkg/calc.py` | `FileWritten { bytes: 75 }`; nested dir created on disk |
| `ExecuteCommand git --version` | `status_code 0`, stdout `git version 2.52.0.windows.1` |
| `ExecuteCommand python hello.py` | `status_code 0`, stdout `PC-HOST-OK 55` (ran the file it had just written) |
| `GitStatus` | `?? hello.py`, `?? pkg/` вЂ” real git seeing the new files |
| `ReadFile hello.py` | content round-trips through the gateway |
| Disk check | `hello.py` (36 B) and `pkg\calc.py` (75 B) exist with the exact written contents |

This is the externally-verifiable counterpart to the in-app phone runs: the gateway protocol genuinely creates real files and executes real commands.

## PCв†”phone pairing bundle (generation verified)

The "connect file" between PC and phone is the pairing bundle generated by the app (`PcGatewayPairingBundle` in core; exported on-device via `PcPairingManager::export_zip`, and also reachable headlessly through `pc_pairing_bundle_probe`). It writes `pairing.json` (gateway id, workspace, bind addr, auth token, transport mode, trusted paths), `deepseek-pc-host.env`, `README.txt`, **`Setup-DeepSeek-PC-Host.{cmd,ps1}`** (one-click firewall + autostart on Windows), manual launchers `start-deepseek-pc-host.{ps1,sh,command}`, optionally embedding the discovered `deepseek-pc-host` binary вЂ” as either a directory or a ZIP. PC projects use the **`Project workspace`** folder beside the bundle (same folder name as the phone sandbox under `files/deepseek-mobile/`).

`cargo test -p deepseek-mobile-core pc_pairing` в†’ **8/8 pass**, including `writes_bundle_zip` and `writes_bundle_zip_with_embedded_host_binary` (which generate a real ZIP and confirm the embedded host binary). The mobile-side `PcPairingManager` is a thin wrapper over this same generator. (The mobile crate's host-side lib test harness hangs because that crate expects an Android runtime to initialize вЂ” an environment limitation, not a pairing defect.)
